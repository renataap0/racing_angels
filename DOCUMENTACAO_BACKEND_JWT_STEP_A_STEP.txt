DOCUMENTAÇÃO DO BACKEND (STEP A STEP) — JWT + Rotas (para gerar versão em JavaScript)

1) Visão geral
Backend: API REST feita com Node.js + Express + TypeScript + Prisma.
Autenticação: JWT (JSON Web Token).
Porta: http://localhost:3333 (padrão em env.port).

2) Como o backend está organizado (pasta e responsabilidade)
- src/server.ts
  Sobe o servidor HTTP e escuta a porta definida em env.

- src/app.ts
  Configura middlewares globais (CORS, express.json), registra rotas e ativa autenticação com JWT.

- src/routes/
  Define endpoints (método HTTP + path) e aponta para controllers.

- src/controllers/
  Recebe request/response, valida entrada quando aplicável e chama services.

- src/services/
  Regras de negócio e acesso ao banco via Prisma.

- src/middlewares/
  authMiddleware.ts: valida JWT e injeta req.user.
  requireRole.ts: (quando usado) restringe por role.
  errorMiddleware.ts: tratamento padrão de erros.

- src/config/
  env.ts: leitura de variáveis e configuração (port, jwtSecret).
  prisma.ts: inicialização do PrismaClient.

- src/schemas/
  Validação (ex.: Zod) para payloads (como login).

- src/utils/
  AppError, asyncHandler e helpers.

3) Fluxo de inicialização (passo a passo)
3.1 src/server.ts
1. Importa { app } de ./app.
2. Importa { env } de ./config/env.
3. Chama app.listen(env.port, ...).

3.2 src/app.ts
1. app.use(cors())
2. app.use(express.json())
3. Rotas públicas:
   - app.use("/api/auth", authRoutes)
   Isso garante que o POST /api/auth/login não precisa de token.
4. Middleware JWT global:
   - app.use("/api", authMiddleware)
   A partir daí, as rotas registradas depois tendem a exigir token.
5. Rotas protegidas registradas em seguida:
   /api/users
   /api/teams
   /api/drivers
   /api/cars
   /api/tracks
   /api/races
   /api/seasons
   /api/season-rounds
   /api/products
   /api/orders
   /api/analytics
   /api/dashboard
6. Fallback 404:
   - qualquer rota não registrada cai em "Rota nao encontrada." (404)
7. errorMiddleware
   - formata respostas de erro.

4) Autenticação JWT (passo a passo)
4.1 Login (rota pública)
Endpoint:
- POST /api/auth/login
Arquitetura:
- route -> controller -> service

Fluxo detalhado:
1. Controller recebe request.body.
2. Valida com loginSchema (Zod).
3. Service authService.login(username, password):
   3.1 Busca usuário no banco por username.
   3.2 Se não existir: AppError 401 "Usuario ou senha invalidos."
   3.3 Compara senha com bcrypt.compare.
   3.4 Se não bater: AppError 401 "Usuario ou senha invalidos."
   3.5 Gera token JWT com:
       - payload: username, role, teamId, driverId
       - subject (sub): user.id
       - expiresIn: "8h"
4. Retorna JSON:
   - { token, user: { id, username, role, teamId, driverId, ... } }

4.2 authMiddleware (proteção das rotas)
Arquivo:
- src/middlewares/authMiddleware.ts

Comportamento:
1. Lê header Authorization.
2. Exige formato: "Bearer <token>".
   - Se não existir: 401 "Token JWT nao informado."
3. Verifica token:
   - jwt.verify(token, env.jwtSecret)
4. Se inválido/expirado:
   - 401 "Token JWT invalido ou expirado."
5. Se válido:
   injeta req.user com:
   - id (Number(decoded.sub))
   - username
   - role
   - teamId (decoded.teamId ?? null)
   - driverId (decoded.driverId ?? null)
6. next() para continuar.

6) Como testar cada rota no Thunder/Postman (passo a passo)
1. Faça login:
   - POST /api/auth/login
   - Salve o campo token retornado.
2. Para QUALQUER rota protegida (todas as rotas sob /api após authMiddleware):
   - Header:
     Authorization: Bearer SEU_TOKEN
3. Teste os endpoints principais:
   - GET /api/dashboard/summary
   - GET /api/analytics/
   - GET /api/analytics/drivers
   - GET /api/analytics/tracks
   - GET /api/analytics/cars
   - GET /api/analytics/rankings
4. Interpretação de erros comuns:
   - 401 Token JWT nao informado / invalido:
     Token ausente ou errado.
   - 403 Usuario sem permissao para esta acao:
     token existe mas role não permite.
   - 404 Rota nao encontrada:
     path/método incorreto.
